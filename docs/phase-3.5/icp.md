# PairSlash ICP

Unless otherwise marked, trigger moments and buying motivations below are [Hypothesis] pending interviews.

## P1

### User archetype

Solo builders who live in Codex CLI or GitHub Copilot CLI, revisit the same repo often, and care about keeping project truth clean.

### Trigger moments

- they return to a repo after a context switch and need to get oriented fast
- they repeat the same terminal workflow often enough to want a safer default
- they realize chat history and ad hoc notes are not reliable project memory

### Buying or adoption motivation

- reduce rework from almost-right AI output
- reduce context rebuild time
- make durable project memory feel safe enough to use regularly
- keep install and setup friction below the pain they are already paying

### What they must see to trust PairSlash

- repo grounding that is fast and noticeably more correct than starting from scratch
- explicit previews before important writes
- no hidden durable writes
- `/skills` as a clear entrypoint instead of magic behavior
- disciplined memory that feels reviewable, not mystical

### Disqualifiers

- mostly work in GUI AI products
- do one-off tasks with no repeated workflow pattern
- do not care about durable project context
- want full autonomy more than trust and control

## P2

### User archetype

[Hypothesis] Tech leads and senior ICs who want a safer, more repeatable terminal-native AI workflow for a small team.

### Trigger moments

- teammates are using AI in inconsistent ways
- repeated almost-right outputs create avoidable review churn
- project context is scattered across chat, PRs, and private notes

### Buying or adoption motivation

- reduce workflow variance across the team
- lower review and cleanup cost
- keep shared project memory disciplined without a heavy process layer

### What they must see to trust PairSlash

- explicit write model with reviewable memory changes
- safe repeated workflows that do not surprise maintainers
- evidence that the product improves team behavior, not just solo convenience

### Disqualifiers

- want a full policy engine first
- need broad enterprise workflow orchestration before proving local value
- do not believe terminal-native AI work should be standardized

## P3

### User archetype

[Hypothesis] DevEx and platform owners exploring how to make terminal-native AI usage safer and easier to adopt across repos.

### Trigger moments

- install friction and workflow inconsistency block broader rollout
- teams ask for safer reuse of proven workflows
- leaders want trust and auditability before they fund broader enablement

### Buying or adoption motivation

- reduce setup friction for approved workflows
- reduce rework caused by almost-right AI behavior
- create a more trustworthy operating model before scaling usage

### What they must see to trust PairSlash

- clear scope limited to Codex CLI and GitHub Copilot CLI
- explicit-write-only behavior for durable memory
- proof that the trust layer changes usage quality, not just docs quality

### Disqualifiers

- expect PairSlash to solve all AI governance needs
- require support for a third runtime
- prioritize breadth over evidence-backed trust gains
