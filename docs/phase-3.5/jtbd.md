## JTBD 1: Understand this repo fast and correctly

### Functional job

Recover the repo's important truth, constraints, and workflow context without wandering through stale files or guessing.

### Emotional job

Reduce the anxiety that the AI is confidently wrong about the codebase.

### Social job

Look credible when re-entering a repo, handing work off, or picking up work from someone else.

### Current workaround

Readme hopping, grep, shell history, chat scrollback, and ad hoc personal notes.

### Switching trigger

The user has paid the cold-start cost often enough that "just re-read the repo" feels wasteful.

### Desired outcome

A faster, more correct repo start with fewer false assumptions.

### Failure mode if PairSlash does this poorly

It produces summaries that sound good but do not improve correctness or save real time.

## JTBD 2: Run a repeated terminal workflow safely

### Functional job

Reuse a workflow in the terminal without hidden side effects and without rebuilding the checklist every time.

### Emotional job

Feel in control when the AI is helpful but not fully trustworthy.

### Social job

Recommend the workflow to a teammate without feeling like you are handing them a trap.

### Current workaround

Manual checklists, shell history, copy-pasted prompts, private scripts, and extra review overhead.

### Switching trigger

The user sees the same almost-right AI mistakes and cleanup steps repeat often enough to want a safer path.

### Desired outcome

A repeatable workflow that lowers rework while keeping control obvious.

### Failure mode if PairSlash does this poorly

It adds ceremony but does not materially increase safety or reduce cleanup.

## JTBD 3: Keep project memory disciplined across sessions and people

### Functional job

Store and reuse durable project truth in a way that is explicit, reviewable, and safe to trust later.

### Emotional job

Stop worrying that memory has silently drifted or been polluted by weak claims.

### Social job

Give collaborators a shared source of truth that is better than folklore or chat fragments.

### Current workaround

Scattered markdown notes, PR comments, chat transcripts, and tribal memory.

### Switching trigger

Context loss or contradiction happens often enough that the user no longer trusts the existing patchwork.

### Desired outcome

Durable project memory that survives time, session changes, and more than one contributor.

### Failure mode if PairSlash does this poorly

It becomes another stale note sink that users stop consulting.
