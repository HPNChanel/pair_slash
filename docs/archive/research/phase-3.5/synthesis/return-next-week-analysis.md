# PairSlash Phase 3.5 Return-Next-Week Analysis

Use this file to answer one question:

Why would a user come back next week instead of trying PairSlash once and
forgetting it?

## Strong Return Drivers

Only count drivers backed by repeated evidence:

| Return driver | Why it matters | Evidence needed | Threshold to call it strong |
|---|---|---|---|
| Same repo re-orientation pain keeps happening | Creates recurring trigger for onboard-repo wedge | concrete return-to-repo incidents + time lost | >= 3 interviews or 2 segments with strong evidence |
| Review/fix loop cleanup keeps repeating | Creates direct rework reduction wedge | almost-right AI rework cases + manual rescue burden | >= 3 interviews with workaround cost |
| Durable project memory prevents repeated context loss | Creates continuity wedge across sessions and people | contradiction/staleness incidents + reliance on scattered notes | >= 3 interviews with trust threshold detail |
| Explicit review and auditability clear trust threshold | Makes disciplined memory usable instead of risky | concrete statement of what must be true before reuse | >= 2 segments with strong trust-threshold evidence |

## One-Time Trial Then Drop Signals

| Drop reason | Why it kills retention | What to look for |
|---|---|---|
| Concept sounds good but no recurring trigger | novelty only | positive language with weak repeat-use signal |
| Setup friction is felt before core value | first-use friction dominates | install complaints with no strong underlying pain |
| Workflow saves thought but not cleanup | no measurable ROI | fast output but same or worse rework burden |
| Trust threshold is not crossed | user will not risk reuse | durable write concern without acceptable guardrail |
| Memory source feels like shelfware | continuity value does not stick | users say they would not check it later |

## Weekly Return Decision Rule

Count a return signal as strong only if all are present:

- recurring trigger exists
- current workaround cost is explicit
- user can name what PairSlash would replace
- trust threshold is clear enough to cross

Downgrade to `weak` or `novelty_only` if:

- the participant only says "would use"
- the trigger is rare
- setup dominates the story
- the behavior change is unclear

## Likely Wedge Candidates to Test Harder

These are candidates, not validated conclusions:

- repo understanding with less cold-start error
- review/fix loop with less almost-right rework
- disciplined project memory with explicit write trust
